# Blocking I/O on virtual threads, JDK 21 floor, TLS in v1

The socket layer (an internal component of the client runtime module, not a separate sbt module) uses plain blocking `Socket`/`SSLSocket` on virtual threads — two per connection (reader feeding the RESP3 parser, writer draining the send queue) — bridged once into `CIO` via `CIO.async` for all backends. This is deliberate, not naïveté: the multiplexed connection model (ADR-0003) means a client holds 1–3 connections (a few per node in cluster mode), so per-connection threads cost nothing, and `SSLSocket` gives battle-tested TLS for free. The alternative — NIO.2 with a hand-driven `SSLEngine` state machine — was rejected as the project's riskiest work item for zero benefit at this connection count. Same architecture as rueidis (goroutines).

Consequences: sage requires **JDK 21+** (the only public surface of this choice); TLS and AUTH/ACL ship in v1, since managed Redis/Valkey (ElastiCache, MemoryDB, Azure, Upstash) all require TLS; nothing socket-related needs to be contributed upstream to kyo-compat before v1 (amends ADR-0001) — the layer can be offered upstream later once production-proven. The choice is invisible in sage's public API and swappable behind the socket layer's internal interface if it ever proves limiting.
