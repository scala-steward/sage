# Multiplexed connection with dedicated-connection escape hatch

Ordinary commands from all fibers are auto-pipelined onto a single multiplexed connection per node, with replies matched FIFO (RESP guarantees in-order replies). Commands that hold per-connection state or block the connection — `BLPOP`-style blocking commands, `WATCH`/`MULTI`/`EXEC` — transparently acquire a dedicated connection from a small on-demand pool. Client-side-caching invalidations ride the multiplexed connection via RESP3 push frames (they are runtime-internal and need no user-side backpressure). User-facing pub/sub instead gets a lazily-created dedicated subscription connection, shared by all subscriptions and auto-resubscribed on reconnect: a slow pub/sub consumer then backpressures only that connection at the TCP level and can never stall command replies on the multiplexed connection. This is the design of the current client generation (lettuce, rueidis, StackExchange.Redis); a plain connection pool was rejected as the previous generation's model — it forfeits automatic pipelining, the main Redis throughput lever. Striping across N multiplexed connections is a possible later tuning knob, not v1.
