# Watchdog detects death by reply silence, not per-command latency

A half-open connection (peer gone with no RST — common in failover and NAT timeouts) never surfaces on a blocking socket read, and TCP keepalive fires in hours, so liveness needs an application-level probe: when the Multiplexed Connection is idle the watchdog injects a `PING` through the normal submit path, and declares the connection dead when an in-flight reply stays unanswered past `pingTimeout`. Because replies are FIFO over one connection, a single slow command at the head blocks every reply behind it — and an app-level PING queues behind it too — so the only robust signal is total reply silence, and `pingTimeout` is a death detector that must be sized above worst-case healthy latency, not a per-command SLA. A killed connection's in-flight commands fail `ConnectionLost(mayHaveExecuted = true)`; a user-facing per-command deadline is deliberately left to each backend's own timeout combinators.
