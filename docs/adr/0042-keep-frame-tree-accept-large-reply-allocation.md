# Keep the Frame tree: accept higher large-aggregate-reply allocation, no streaming decode path

The Core parser materializes a generic `Frame` tree that reply decoders then consume (ADR-0001). On large aggregate replies this costs roughly twice the allocation of a client that decodes the wire straight into the typed result: the benchmark harness measures sage at ~714 KB/op on `HGETALL` of 1000 fields versus Lettuce's ~405 KB/op (1.76Ã—). We are **keeping the `Frame` model and accepting that cost**; we will not add a streaming, decoder-ahead path to eliminate the intermediate tree.

## Why

The gap is fully explained and intrinsic to the model. Lettuce decodes wire â†’ `String` in a **single copy**: it knows the typed decoder (`CommandOutput`) *before* it parses, because replies are FIFO, and reads bytes straight from a retained network buffer (`Unpooled.wrappedBuffer(bytes).toString(charset)`) into a pre-sized map â€” so per key/value pair it allocates essentially just the result. sage pays **two** copies per pair: the parser copies bytes out of its reused read buffer into a `Frame.BulkString(Bytes(IArray))` (the buffer is overwritten on the next socket read, so the copy is mandatory if the frame is to outlive `feed`), and the codec then copies again into the `String`. The extra copy, plus the `Bytes`/`Tuple2`/`Vector` wrappers and the `Frame.Map` tree, is the whole difference. There is no partial win available within the `Frame` model â€” each of the two copies is individually mandatory once a `Frame` exists.

Closing the gap therefore means adopting the industry-standard decode-ahead model (Lettuce, go-redis, redis-py, Jedis all prime the typed decoder before parsing and stream wire values into it, with no generic tree). sage cannot adopt it wholesale, because the `Frame` tree is load-bearing for features that require a materialized reply: `EVAL`/`FCALL` return raw `Frame`s to the user (ADR-0033, a public contract), client-side caching stores `Frame`s and re-decodes them per waiter (ADR-0028), `EXEC`/Pipeline decode per-position over a materialized `Vector[Frame]` (ADR-0019/0020), and lenient stream-introspection decoding inspects frames (ADR-0024). The only viable shape is a *hybrid* â€” a streaming fast path plus the `Frame` path retained for those cases â€” which makes the Core carry two decode models in its hottest code and couples the parser to the runtime's FIFO queue and to out-of-band push disambiguation (the `>` type byte), breaking the clean sans-IO "parser emits `Frame`s" boundary (ADR-0001).

That is a large, high-risk change to the Core's central abstraction for a narrow, single-workload gain. Everywhere else sage is at parity or better â€” on small replies sage-ox allocates *less* than Lettuce (â‰ˆ1084 vs 1445 B/op) â€” and large collection reads are a minority of real traffic. The cost is not worth the complexity today.

- Trigger to reconsider: a concrete, measured need for low-allocation large-reply reads (a production workload dominated by big `HGETALL`/`MGET`, or a head-to-head requirement against Lettuce on that path). Revisiting then means designing the hybrid streaming decoder, not a tweak.

## Considered Options

- **Build the hybrid streaming decode path** (decoder primed ahead for hot commands, `Frame` retained for raw/cache/EXEC) â€” rejected now: high complexity and broad blast radius across every decoder, and it erodes the sans-IO boundary, all for a single-workload allocation win with no current production driver.
- **A within-`Frame` micro-optimization** (e.g. removing the double copy) â€” rejected as impossible: the double copy is the `Frame` materialization itself; neither copy is removable while a `Frame` is produced.
