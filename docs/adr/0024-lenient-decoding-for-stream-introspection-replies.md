# Lenient field-presence decoding for stream introspection replies

The `XINFO STREAM`/`STREAM FULL`/`GROUPS`/`CONSUMERS`, `XPENDING`, and `XAUTOCLAIM` decoders read their reply maps by known key, ignoring unrecognized fields and treating version-specific fields (`entries-read`, `lag`, the `FULL` nesting, …) as optional — a deliberate exception to the strict-primitive-codecs rule, which fails a decode on anything but the type's canonical form. The reason the rules differ: a primitive's wire form is fixed and a near-miss is a real bug worth surfacing, whereas an introspection reply's field set legitimately grows and varies across the server matrix sage tests against (Redis 6/7/8 and Valkey diverge — `entries-read`/`lag` only exist from 7.0, `FULL` keeps gaining fields), so requiring an exact field set would make a correct client break against a perfectly healthy server. Strict decoding here buys nothing — there is no canonical form to enforce — and costs forward and cross-server compatibility, which is why these replies are typed (the values stay strongly typed; `Frame` is never exposed) but decoded leniently on presence. The escape hatch the dynamically-typed clients use (redis-py returns a raw `dict`, ioredis a raw array) is unavailable by design: sage has no user-facing untyped reply container, so the typed-but-lenient model is how it gets both the ergonomics of typing and the robustness those clients get from dicts.
