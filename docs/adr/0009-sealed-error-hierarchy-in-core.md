# Single sealed error hierarchy, defined entirely in the core

All sage failures form one sealed exception hierarchy so users can handle classes of failure exhaustively. Scala's `sealed` confines a hierarchy to one file, yet the cases span both modules' concerns: the core produces protocol, decode and server errors, while the runtime raises connection-lost (with its may-have-executed flag), not-connected and timed-out. The whole hierarchy therefore lives in `sage-core`, in one file; the runtime only raises cases the core defines. This works because every failure case is pure data — defining `NotConnected` requires no I/O — and exceptions are dependency-free, so the zero-dependency constraint (ADR-0004) is untouched. The rejected alternative — a pure error ADT in the core wrapped into exceptions by the client — produced two parallel failure vocabularies and an unwrapping layer users would have to learn; an unsealed base class was rejected outright as it forfeits exhaustive matching.

`ServerError` carries the leading error token as a structured `code` (`ServerError(code, detail)`), parsed once from the wire message by `ServerError.of`, so callers branch on `case ServerError("WRONGTYPE", _)` instead of `message.startsWith("WRONGTYPE")` — the runtime itself does this for `NOPROTO`/`READONLY`. The code space is open (Redis defines hundreds of prefixes), so it stays a `String` field rather than a closed set of subtypes that would force an arbitrary cutoff and a catch-all; the sealed hierarchy still matches exhaustively at the `ServerError` case.
