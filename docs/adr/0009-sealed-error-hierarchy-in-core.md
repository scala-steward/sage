# Single sealed error hierarchy, defined entirely in the core

All sage failures form one sealed exception hierarchy so users can handle classes of failure exhaustively. Scala's `sealed` confines a hierarchy to one file, yet the cases span both modules' concerns: the core produces protocol, decode and server errors, while the runtime raises connection-lost (with its may-have-executed flag), not-connected and timed-out. The whole hierarchy therefore lives in `sage-core`, in one file; the runtime only raises cases the core defines. This works because every failure case is pure data — defining `NotConnected` requires no I/O — and exceptions are dependency-free, so the zero-dependency constraint (ADR-0004) is untouched. A side benefit: runtimes built on the JS/Native core share the same failure vocabulary. The rejected alternative — a pure error ADT in the core wrapped into exceptions by the client — produced two parallel failure vocabularies and an unwrapping layer users would have to learn; an unsealed base class was rejected outright as it forfeits exhaustive matching.
