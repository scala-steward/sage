# One client type; topology is configuration

There is a single `SageClient` type; standalone vs cluster is decided by configuration, not by distinct client types (rejected: lettuce-style `RedisClient`/`RedisClusterClient` split, which infects application code with deployment topology). In cluster mode, Pipelines that cross slots are transparently split per node, run in parallel, and merged in submission order — Pipelines never promised atomicity. Transactions fail fast with a cross-slot error, since `MULTI`/`EXEC` is genuinely single-node. The streaming `scanAll` stays topology-portable: a SCAN cursor is node-local, so in cluster mode it walks every slot-owning master in turn — each with its own cursor — concatenating the pages, while the same call scans the one node standalone (a reshard mid-scan can still miss or duplicate keys, as on any client). Genuinely cluster-only operations (per-node commands) live in a clearly marked sub-API.
