# Graceful drain on close; fail-fast only on unexpected loss

`close` stops accepting new commands (which then fail `NotConnected`) and awaits in-flight replies up to `closeTimeout` before force-closing, so a controlled shutdown lets already-sent commands collect their real replies instead of a pessimistic `ConnectionLost(mayHaveExecuted = true)`. This refines ADR-0006: fail-fast belongs to *unexpected* connection loss, where the socket is gone and a possibly-executed command must not be silently retried; a deliberate close is different because the connection is still alive, and draining is more precise, not less. The wait is bounded — unlike rueidis's unbounded `Close` — so a wedged server can't hang shutdown forever, and survivors past the timeout fall back to `ConnectionLost(true)`. Both of sage's closest architectural peers drain on close: Lettuce via a quiet-period plus timeout, rueidis by awaiting all pending calls.
