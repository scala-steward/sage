# Cluster topology and routing live in the pure core, as data and total classification

The cluster slot engine — CRC16 slot hashing with hash-tag extraction, the topology model, the `(topology, command) → node` routing decision, redirect parsing, and the Pipeline split/merge plan — lives entirely in the sans-IO core (ADR-0001), holding the topology as **data** and expressing every routing decision as a **total classification** that never raises, never picks a connection, and never decides a retry. The core gains `Node(host, port)`, `Shard`, and `ClusterTopology` even though an address looks like I/O territory, because a slot's owner is a pure fact about the keyspace, not an effect: given the same topology and command, the target is determined, so it belongs to the pure layer that is cheap to test against synthetic topologies (PRD story 36). ADR-0016 already put a routing *hint* (`Execution`) on the command and deferred the rest; this is the rest, and it stays pure for the same reason.

`ClusterTopology.route(cmd)` returns a sealed `Route` — `ToNode(node, slot) | Keyless | Unowned(slot) | CrossSlot(slots)` — rather than `Either[error, Node]`. The four outcomes demand four different runtime reactions (send / pick any node / refresh topology / fail), and only `CrossSlot` ever becomes a `SageException`; folding the other three into an error channel would misclassify a keyless command or a transient resharding gap as a failure. The engine *classifies*; the runtime owns the policy of what to raise, when to refresh, and which node serves a keyless command. For the same reason `ClusterTopology.from(shards)` is total — an uncovered slot yields `nodeForSlot = None → Unowned`, the runtime's refresh signal, not a construction error — and `Redirect.parse` returns `Option`, leaving `Reply.run` untouched: the cluster runtime try-parses a `ServerError` message and falls through on `None`, so the standalone reply path is unchanged.

## Considered Options

- **Topology and routing in the pure core (chosen)** — slot ownership is a pure function of topology + key; testable with no server, no sockets, consistent with the sans-IO boundary.
- **Routing in the runtime, only hashing in core** — keeps `Node` out of core, but scatters the routing rule across four per-backend runtimes and forfeits the synthetic-topology tests the acceptance criteria require.
- **`Either[RoutingError, Node]` instead of a `Route` ADT** — fewer cases, but conflates "no keys" and "slot not owned yet" with genuine errors and pushes the keyless-node choice into the engine.

## Consequences

- The `SplitPlan` reports per-position `Unowned`/`CrossSlot` in a `rejected` bucket instead of failing the whole batch, inheriting ADR-0019's per-position truth: one cross-slot command in a Pipeline fails its own slot, not the others. Original indices are preserved per node group, so the runtime's merge "in submission order" (ADR-0007) is a scatter back into an N-slot array, not a separate merge structure.
- Keyless commands are surfaced (`Route.Keyless`, the `SplitPlan.keyless` bucket), not assigned a node, because "any node" is a runtime policy — reuse an existing connection, least-loaded — not a pure fact.
- Shard pub/sub adds no channel-specific API: a channel is bytes fed to the same `Slot.of`, so the runtime composes `nodeForSlot(Slot.of(channel))`. "Channels hash like keys" is then literally true and tested through `Slot.of`.
- A `Redirect` carries a `Node` target; an empty host in the reply (`MOVED 3999 :6381`) becomes a `Node` with an empty host the runtime resolves to the current connection's IP. The engine does not act on the `Moved`/`Ask` distinction — it only records it.
