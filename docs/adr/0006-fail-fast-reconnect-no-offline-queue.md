# Fail-fast on disconnect; sage does not own retry policy

On connection loss the runtime always auto-reconnects (configurable backoff) and restores connection-owned state (pub/sub subscriptions, client tracking). Commands are not protected: in-flight commands fail with `ConnectionLost` (flagged ambiguous — they may have executed server-side), and commands submitted while disconnected fail fast with `NotConnected`. There is no offline command queue (lettuce/ioredis style) and no automatic retry: retrying a possibly-executed non-idempotent command (e.g. `INCR`) silently corrupts data, and every target backend already has first-class retry combinators — sage exposes precise failures and lets users own the policy. Safe auto-retry of provably-unsent or idempotent commands (rueidis style) is a possible later refinement.
