# The Listener SPI is a synchronous `Unit` callback in the Core, fed off a bounded lossy queue

The observability SPI is a `SageListener` with one `def onEvent(event: SageEvent): Unit`, and both it and the sealed `SageEvent` hierarchy live in `sage-core`, not in a backend. A `Unit` callback is effect-agnostic: it is registered once in the shared `SageConfig` and behaves identically across all four Backends, with no per-backend lowering, and an integration module (a future `sage-otel`) depends only on the zero-dependency Core rather than binding to ZIO, cats-effect, Kyo, or Ox. The rejected alternative was a backend `F[Unit]` callback, which would force the SPI into the client layer, parameterize it per Backend, and entangle event delivery with the user's effect runtime and its own blocking and cancellation semantics. The cost is that a listener wanting to do async I/O must hand off to its own executor — acceptable, because the SPI's job is to emit, and what the listener does with an event is its own concern.

Events are not delivered inline. Command completions and connection-loss are detected on the Multiplexed Connection's reader thread, which processes every FIFO reply for the whole connection; invoking a listener there would let a slow listener stall every subsequent command's reply and a throwing one corrupt the reply loop. Instead the hot path does a single non-blocking enqueue onto a bounded queue (1024 deep), and one shared daemon dispatcher thread per client drains it and fans each event to every listener, each call wrapped to swallow throws. A listener slow enough to fill the queue causes its events to be **dropped** (drop-newest), silently. This is the deliberate trade required by "a slow or throwing listener cannot block or break command execution": the only relief that never blocks the producer is to shed events, so delivery is lossy under overload rather than lossless-but-blocking — the opposite choice from pub/sub (ADR-0025), where blocking a dedicated connection is acceptable but dropping a Message is not. When no listener is registered the emit sink is a shared no-op and no dispatcher thread starts, so the runtime pays nothing.

## Consequences

- Event delivery is at-most-once and globally ordered (one queue, one dispatcher). A pathological listener delays delivery for the others but never blocks command execution, and never affects another listener's correctness (throws are isolated).
- Dropped events under overload are not surfaced in v1 (sage has no logging facility and there is no read API). A healthy listener never fills a 1024-deep queue; one that does is misbehaving, and dropping is the correct backpressure-free relief.
- The queue depth is an internal constant, not a `SageConfig` knob — it can be promoted to configuration backward-compatibly if a real need appears.
- `SageEvent` carries only a command's name, never its arguments or payloads, so secrets (`HELLO`/`AUTH` credentials, user values) never reach a listener.
